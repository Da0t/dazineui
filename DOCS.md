13. Build Order and Approach
The build proceeds in a specific order designed to validate the system end-to-end before scaling up.
Phase one: Foundation. Scaffold the Next.js project with TypeScript, Tailwind, and the core dependency list. Set up the folder structure exactly as specified above. Write the first version of CLAUDE.md, AGENTS.md, and .cursor/rules/motion-rules.mdc — even though no primitives exist yet. The CLAUDE.md describes what will be there and what the AI should do once primitives are added. Set up lib/tokens/ with the parser and defaults. Set up lib/shaders/ with the noise, easing, and color utilities (or stubs to be filled in). Set up lib/hooks/ with the shared hooks. The result of phase one is a fully-scaffolded project with no primitives but a complete frame around where they'll go.
Phase two: The first primitive. Build FlowGradient end-to-end. FlowGradient is chosen first because it has the highest visual impact for the lowest implementation complexity. A working shader-based gradient mesh is dramatic on first sight and serves as a great proof-of-concept demo. Build the component code in lib/motion/flow-gradient/index.tsx. Write the shader in shader.ts. Define presets in presets.ts. Type the props in types.ts. Generate or render a PREVIEW.png. Write the MANIFEST.md carefully — this is the AI-facing document that determines whether the system actually works. Build the demo page at app/primitives/flow-gradient/page.tsx showing all four presets.
Phase three: Validation. Before building any more primitives, validate that the system works end-to-end. Open a fresh Claude Code session in the project. Prompt: "Add a hero section with a premium-feeling animated background." Observe what Claude does. Does it read CLAUDE.md? Does it find FlowGradient in the MANIFEST? Does it import it correctly? Does it pass appropriate props? If yes, the system is working. If no, the CLAUDE.md and MANIFEST need refinement before scaling. This validation step is critical — it's better to discover the AI isn't reaching for the primitive after building one than after building six.
Phase four: The remaining primitives. Build HeroScene, ScrollScene, MagneticCursor, InteractiveText, and ParticleField in that order. HeroScene is next because it's the showpiece, and its three presets (gradient-orb, wireframe-object, particle-cloud) cover the most common hero patterns. ScrollScene follows because scroll-linked motion is the second-most-requested effect. The micro-interaction primitives — MagneticCursor and InteractiveText — come after the major scenes because they layer onto existing content. ParticleField is last because it's somewhat redundant with HeroScene's particle-cloud preset and serves a narrower use case.
After each primitive ships, run the same validation: prompt Claude with an ambiguous request that should match the new primitive, observe whether the system reaches for it correctly, and refine the CLAUDE.md or MANIFEST if not.
Phase five: Integration and polish. With all six primitives shipped, focus on integration polish. Make sure all primitives can be composed without visual conflicts. Verify that the design-token system flows through correctly when a DESIGN.md is present. Build the home page demo (app/page.tsx) that composes multiple primitives into a single showcase landing page. Write the README with installation instructions, primitive overviews, and acknowledgments. Add a CONTRIBUTING.md for future community contributions.
Phase six: Public release. Push to GitHub as a public repository under the MIT license. Publish the npm package for the CLI installer. Set up a simple documentation site (the demo pages double as documentation). Quietly use it on personal projects and hackathons. Iterate based on real usage.
There is no marketing phase. The project goes public when it's actually good — when the author would genuinely reach for it during their own work. No Twitter announcement, no Product Hunt launch, no Hacker News submission until the tool has proven itself in the author's own hackathon workflow. Premature marketing of an unfinished tool damages its credibility permanently.

14. Working with Claude Code
The development workflow uses Claude Code in Visual Studio Code as the primary AI assistant. Claude Code's agentic file-handling capabilities, multi-file context, and direct terminal access make it well-suited to a project that spans many files and requires careful coordination between primitive code, type definitions, manifests, and instruction files.
Starting each session. Begin every new Claude Code session by ensuring BRIEF.md and CLAUDE.md are in context. Claude Code automatically reads CLAUDE.md. Reference BRIEF.md explicitly in the opening prompt of major sessions: "Read BRIEF.md for project context, then let's continue with [task]."
Scoping each session. Don't ask Claude Code to do too much at once. A good session has a focused goal: "Scaffold the project structure," "Build the FlowGradient component," "Write the MANIFEST and demo page for FlowGradient." Scope sessions narrowly enough that Claude can complete them in one continuous flow without losing track of context.
Reviewing output. After Claude Code completes a session, review the diff before accepting. The author may not know every line of code intimately, but should at least skim for obvious red flags — unexpected dependencies, files in wrong locations, deviations from the established structure. If something looks off, ask Claude to explain it rather than blindly accepting. The goal over time is to understand the entire codebase, even if Claude wrote most of it.
Iterating on primitives. Building a primitive is rarely one-shot. Expect three to five rounds of refinement per primitive: initial implementation, visual polish, performance check, preset tuning, manifest writing. Use Claude Code's diff workflow to keep changes small and reviewable.
Managing the CLAUDE.md. The CLAUDE.md is the single most important file. Edit it deliberately and review it carefully. When a rule changes, consider whether the change applies broadly or only to a specific primitive — if specific, it might belong in a MANIFEST instead. Avoid letting the CLAUDE.md grow indefinitely; it should stay focused on rules that genuinely apply project-wide.
Testing the system. Periodically, run a fresh Claude Code session with a "naive" prompt — something like "add a hero section" — and observe how the system handles it without specific guidance. This catches drift in the CLAUDE.md and reveals primitives that aren't being matched correctly. Treat these sessions as integration tests.
Cursor as a secondary check. Even though primary development is in Claude Code, periodically open the project in Cursor (or another AI tool) and run the same naive prompts. This validates that the .cursor/rules/motion-rules.mdc mirror is working and that the system is truly tool-agnostic.

15. Glossary of Terms
For clarity, here are the key terms used throughout this document, defined in plain language.
AI coding assistant. A tool like Claude Code, Cursor, GitHub Copilot, Windsurf, or Codex that helps developers write code by understanding natural-language prompts and generating or modifying source files. These tools are distinct from no-code generators like Lovable or Bolt — they work alongside human developers in an IDE, not as standalone site builders.
CLAUDE.md. A markdown file in a project's root directory that Claude Code (and similar tools) automatically reads at the start of every session. It contains project-specific instructions, rules, and context. AGENTS.md serves the same role for Codex and other agents. .cursor/rules/*.mdc serves the same role for Cursor.
DESIGN.md. An emerging open format for defining a project's visual design system in a single markdown file. Pioneered by Google Stitch and now supported by libraries like freedesignmd, designmd.app, and getdesign.md. Contains color tokens, typography, spacing, components, and motion specifications. AI assistants read it to apply consistent styling.
MANIFEST.md. A project-specific file format used in this codebase. Each primitive has its own MANIFEST.md describing what the primitive looks like, when to use it, what props it accepts, and what prohibitions apply. Designed to be read by AI assistants.
Primitive. A pre-built, self-contained motion or 3D component shipped as part of this library. Examples: FlowGradient, HeroScene, ScrollScene. Each primitive is hand-crafted by humans (or with significant human curation), polished to ship-quality, and designed to be composed with other primitives.
React Three Fiber (R3F). A React renderer for Three.js. Allows developers to write 3D scenes as declarative React components rather than imperative Three.js code. The foundation of all 3D primitives in this library.
Three.js. The most widely-used JavaScript library for 3D graphics on the web. Wraps WebGL with a higher-level API. R3F builds on top of Three.js.
Shader. A small program written in GLSL (OpenGL Shading Language) that runs on the GPU and computes color values for each pixel (fragment shader) or position values for each vertex (vertex shader). Shaders are how high-end web animations achieve smooth, performant visual effects.
Post-processing. A rendering technique where the entire scene is rendered, then passed through a series of additional shader passes that add effects like bloom (glow on bright areas), chromatic aberration (color fringing at edges), depth-of-field (blur based on distance), and tonemapping (color grading). Post-processing is what separates "WebGL demo" from "production cinematic site."
Token. A named design value — a color, font, spacing unit, timing value — defined in a DESIGN.md and consumed by primitives. Tokens enable the same primitive to render differently across projects with different design systems.
Preset. A named configuration of a primitive. FlowGradient has presets like aurora, sunset, electric, monochrome. Each preset is a pre-tuned set of prop values that produces a specific visual character. Presets are the AI's preferred way to configure primitives.
MCP (Model Context Protocol). An open protocol that lets AI assistants connect to external tools and data sources. The shadcn/ui MCP server, for example, lets Claude Code browse and install shadcn components. This project does not ship an MCP server in v1 but may in future versions.
Slop. Informal term for the generic, low-quality output that AI coding assistants tend to produce by default — basic gradients, default fade-ins, hover:scale-105, system fonts. The aesthetic this project exists to eliminate.

16. Learning Path Reference
If foundational learning is needed at any point, this is the canonical order. None of this is mandatory if the relevant skills are already present, but it's included for completeness.
JavaScript fundamentals first. Variables, functions, async/await, array methods, ES modules, destructuring, the spread operator. The free MDN Web Docs JavaScript guide is the canonical reference. freeCodeCamp's JavaScript Algorithms and Data Structures certification covers everything needed.
React fundamentals next. Components, props, state, useEffect, useState, useRef, useContext, useMemo. The official React documentation at react.dev is exceptional — the "Learn React" track is the right starting point. Build two or three small projects (todo app, weather app, simple game) to internalize the patterns.
TypeScript light introduction. Enough to read and write type annotations, define interfaces and unions, and understand generics at a basic level. The official TypeScript Handbook's "Everyday Types" and "Object Types" sections are sufficient. Advanced features (conditional types, mapped types, infer) are not required for this project.
Next.js basics. The official nextjs.org/learn course covers the App Router, server components, client components, routing, and deployment. A focused weekend of study is enough to build productively.
Tailwind CSS. The official Tailwind documentation is exceptional and short. Read the core concepts page and the utility-first fundamentals page. The rest is reference material as needed.
Three.js fundamentals. Bruno Simon's Three.js Journey is the canonical paid course (approximately $95) and is genuinely excellent. The official Three.js documentation and the Three.js Fundamentals website (threejsfundamentals.org) are free alternatives. Focus on understanding scenes, cameras, lights, geometries, materials, and the render loop.
React Three Fiber. The official R3F documentation at r3f.docs.pmnd.rs. Read the introduction, the canvas guide, and skim the API. Then study the @react-three/drei Storybook to see what helpers are available. Yuri Artyukh's YouTube channel ("Yuri Codes") has excellent advanced tutorials. Wawa Sensei's R3F YouTube series is great for beginners.
GLSL and shaders. This is the deepest topic and the most optional. For this project, custom shaders are limited to what's already in lib/shaders/, and most primitives compose existing shader utilities rather than writing new ones. If interest in shaders grows, The Book of Shaders (thebookofshaders.com) is the canonical free resource. Inigo Quilez's shader articles at iquilezles.org are advanced but illuminating.
A realistic learning path from "new to web development" to "ready to build this tool well" takes three to six months of focused effort. Anyone with existing React fluency can probably get there in one to two months by adding R3F and Three.js to their skill set. Anyone already comfortable with R3F can start building immediately.

The project's success is not measured by GitHub stars, downloads, or social media attention. It's measured by a single question: does it make the author's own hackathon and side-project work meaningfully better? If yes, every additional user is a bonus. If no, no amount of external metrics matters.
